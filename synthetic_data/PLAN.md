# Synthetic Data Generation Plan

## Goal

Generate ~10,000 synthetic e-commerce customer support tickets for fine-tuning an NLP classification model. Each row will have an LLM-generated ticket text and a deterministic **priority label** (low / medium / high / urgent) derived from the row's structured metadata.

The [rjac/e-commerce-customer-support-qa](https://huggingface.co/datasets/rjac/e-commerce-customer-support-qa) dataset is used only to inform the **label taxonomy** — we do not use its text. All ticket text is generated by an LLM prompted with structured metadata combinations.

---

## Core Approach

1. Define all categorical label columns and their possible values
2. Generate a sampled cross-product of those combinations (~10,000 rows)
3. Derive `priority` deterministically from `customer_sentiment` × `issue_complexity`
4. Prompt an LLM with each row's metadata to generate the `ticket_text`
5. Export to CSV

The LLM's only job is natural language generation. All structured columns are known before any LLM call is made.

---

## Output Schema

| Column | Type | Description |
|---|---|---|
| `ticket_id` | int | Unique row identifier |
| `ticket_text` | str | LLM-generated customer message (the ticket body) |
| `issue_area` | str | Top-level issue category |
| `issue_category` | str | Mid-level category |
| `issue_sub_category` | str | Granular sub-category |
| `customer_sentiment` | str | neutral / negative / frustrated |
| `issue_complexity` | str | less / medium / high |
| `product_category` | str | e.g. Electronics, Appliances |
| `product_sub_category` | str | e.g. DSLR Camera, Water Purifier |
| `priority` | str | **Target label**: low / medium / high / urgent |

## Priority Label Logic

Priority is derived deterministically from `customer_sentiment` × `issue_complexity`:

| Sentiment | Complexity | Priority |
|---|---|---|
| neutral | less | low |
| neutral | medium | medium |
| neutral | high | medium |
| negative | less | medium |
| negative | medium | high |
| negative | high | high |
| frustrated | less | high |
| frustrated | medium | high |
| frustrated | high | urgent |

~7% of rows will have priority randomly perturbed by ±1 level to simulate real-world label noise.

### Target Distribution

| Priority | Target % | Target Count |
|---|---|---|
| low | ~20% | ~2,000 |
| medium | ~35% | ~3,500 |
| high | ~30% | ~3,000 |
| urgent | ~15% | ~1,500 |

## LLM Generation Strategy

### Prompt Design

Each LLM call passes the row's structured metadata and asks for only the ticket text:

```
You are simulating a customer contacting e-commerce support.

Write a single customer support message of 3-5 sentences (75-150 words) with these characteristics:
- Issue area: {issue_area}
- Issue category: {issue_category}
- Sub-category: {issue_sub_category}
- Product: {product_sub_category} ({product_category})
- Customer sentiment: {customer_sentiment}
- Issue complexity: {issue_complexity}

Write only the customer's message. Do not include agent replies, greetings from the agent, or any metadata.
The priority signal (urgency, frustration, complexity) should be clear from the language itself — not stated explicitly.
```

**Why 75–150 words:** DistilBERT has a 512-token limit, but priority classification signal lives in the first few sentences. Shorter inputs reduce noise, speed up training, and match the length of real support tickets. Texts under 20 tokens lack enough signal; texts over 200 tokens add padding without benefit.

### Batching

Each request asks the LLM to return a **JSON array of 10 tickets** for 10 different metadata combinations passed in a single prompt. This reduces ~10,000 calls to ~1,000 calls.

**Why batch size 10, not 20:** Quality degrades noticeably in larger batches — later items become shorter, more repetitive, and the model loses track of the specific metadata for each scenario (e.g., blurring "frustrated/high" and "negative/medium" into the same tone). JSON formatting errors also increase with batch size. 10 is a practical sweet spot between cost and quality. Batch size 1 would produce the most consistent output but at 10x the cost and call volume.

```
Given the following 10 customer support scenarios, write one ticket message for each.
Return a JSON array of 10 strings in the same order.

Scenarios:
1. issue_area=Order, sub_category=Delivery delay, sentiment=frustrated, product=Laptop...
2. issue_area=Warranty, sub_category=Claim process, sentiment=neutral, product=Refrigerator...
...
```

### Model

**Recommended: `claude-sonnet-4-6` (Claude Sonnet) at batch size 10.**

Smaller models (e.g., Haiku) are reliable at batch size 1 but degrade faster at larger batch sizes — more likely to ignore specific metadata, produce repetitive phrasing, or return malformed JSON. For this task, quality matters: the `ticket_text` must implicitly encode `customer_sentiment` and `issue_complexity` because those are the priority signal. Noisy text generation at data creation time produces noisy training labels that are hard to fix later.

Sonnet handles instruction-following at batch size 10 reliably, produces more linguistically diverse outputs, and keeps JSON structure intact. Estimated cost: ~1,000 API calls.

**Alternative: Gemini** (e.g., `gemini-2.5-pro`) should behave similarly to Sonnet at batch size 10, but this is less tested for this specific task. If using Gemini, validate empirically: run 50 rows at batch sizes 1, 5, and 10 and check whether sentiment/complexity cues are clearly present and texts are non-repetitive before committing to full generation.

## Generation Pipeline

### Step 1 — Build the Combinations Grid (`build_grid.py`)
- Define all taxonomy values as Python lists/dicts
- Generate the full cross-product using `itertools.product`
- Sample down to exactly 10,000 rows (stratified by priority to hit target distribution)
- Assign `ticket_id` and `priority` to every row
- Save as `data/grid.csv` (the "skeleton" — no ticket text yet)

### Step 2 — Generate Ticket Text (`generate.py`)
- Read `data/grid.csv`
- Chunk rows into batches of 10
- For each batch, build the prompt and call the Claude API
- Parse the returned JSON array and write `ticket_text` back to each row
- Save incrementally to `data/tickets_raw.csv` (so progress is not lost on failure)

### Step 3 — Validate & Clean (`validate.py`)
- Drop any rows where `ticket_text` is empty or malformed
- Assert no exact duplicate `ticket_text` values
- Check priority class distribution matches targets (warn if off by >5%)
- Save final output to `data/tickets.csv`

## Downstream Use

`tickets.csv` feeds into `scripts/train_model.py`. The model is trained to predict `priority` from `ticket_text`.

Fine-tuning target: `distilbert-base-uncased` via Hugging Face `transformers` + `Trainer` API, replacing the current TF-IDF + Logistic Regression baseline.
