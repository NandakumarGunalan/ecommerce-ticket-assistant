## Goal

Generate exactly 11,340 synthetic e-commerce customer support tickets for fine-tuning an NLP classification model. Each row will have an LLM-generated ticket text and a deterministic **priority label** (low / medium / high / urgent) derived from the row's structured metadata.

The [rjac/e-commerce-customer-support-qa](https://huggingface.co/datasets/rjac/e-commerce-customer-support-qa) dataset is used only to inform the **label taxonomy** — we do not use its text. All ticket text is generated by an LLM prompted with structured metadata combinations.

---

## Core Approach

1. Define all categorical label columns and their possible values
2. Generate the full cross-product of those combinations (exactly 11,340 rows: 70 issue triples × 18 product combos × 3 sentiments × 3 complexities)
3. Derive `priority` deterministically from `customer_sentiment` × `issue_complexity`
4. Prompt an LLM with each row's metadata to generate the `ticket_text`
5. Export to CSV

The LLM's only job is natural language generation. All structured columns are known before any LLM call is made.

## Priority Label Logic

Priority is derived deterministically from `customer_sentiment` × `issue_complexity`:

| Sentiment | Complexity | Priority |
|---|---|---|
| neutral | less | low |
| neutral | medium | medium |
| neutral | high | medium |
| negative | less | medium |
| negative | medium | high |
| negative | high | high |
| frustrated | less | high |
| frustrated | medium | high |
| frustrated | high | urgent |

~7% of rows will have priority randomly perturbed by ±1 level to simulate real-world label noise.

### Target Distribution

| Priority | Target % | Target Count |
|---|---|---|
| low | ~20% | ~2,268 |
| medium | ~35% | ~3,969 |
| high | ~30% | ~3,402 |
| urgent | ~15% | ~1,701 |

## LLM Generation Strategy

### Prompt Design

Each LLM call passes the row's structured metadata and asks for only the ticket text:

```
You are simulating a customer contacting e-commerce support.

Write a single customer support message of 3-5 sentences (75-150 words) with these characteristics:
- Issue area: {issue_area}
- Issue category: {issue_category}
- Sub-category: {issue_sub_category}
- Product: {product_sub_category} ({product_category})
- Customer sentiment: {customer_sentiment}
- Issue complexity: {issue_complexity}

Write only the customer's message. Do not include agent replies, greetings from the agent, or any metadata.
The priority signal (urgency, frustration, complexity) should be clear from the language itself — not stated explicitly.
```

**Why 75–150 words:** DistilBERT has a 512-token limit, but priority classification signal lives in the first few sentences. Shorter inputs reduce noise, speed up training, and match the length of real support tickets. Texts under 20 tokens lack enough signal; texts over 200 tokens add padding without benefit.

### Batching

Each request asks the LLM to return a **JSON array of 5 tickets** for 5 different metadata combinations passed in a single prompt. This reduces ~10,000 calls to ~2,000 calls while still keeping generation manageable.

**Why batch size 5:** We decided to use batch size 5 after validation because it is a safer balance between throughput and output quality. Compared with larger batches, it lowers the risk of repetitive phrasing, metadata drift across items, and malformed JSON. Batch size 1 would provide the strongest per-example control, but at much higher call volume and runtime. Batch size 10 is more efficient on paper, but we are moving forward with 5 to prioritize cleaner supervision signals for training.

```
Given the following 5 customer support scenarios, write one ticket message for each.
Return a JSON array of 5 strings in the same order.

Scenarios:
1. issue_area=Order, sub_category=Delivery delay, sentiment=frustrated, product=Laptop...
2. issue_area=Warranty, sub_category=Claim process, sentiment=neutral, product=Refrigerator...
...
```

### Model

**`gemini-2.5-pro` at batch size 5.**

For this task, quality matters: the `ticket_text` must implicitly encode `customer_sentiment` and `issue_complexity` because those are the priority signal. Noisy text generation at data creation time produces noisy training labels that are hard to fix later.

Gemini 2.5 Pro handles instruction-following at batch size 5 reliably, produces linguistically diverse outputs, and keeps JSON structure intact. Estimated cost: ~2,268 API calls for 11,340 rows. Validate small samples first, then scale up only after checking sentiment/complexity cues, repetition, and JSON reliability.

## Generation Pipeline

### Step 1 — Build the Combinations Grid (`build_grid.py`)
- Define all taxonomy values as Python lists/dicts
- Generate the full cross-product using `itertools.product` (70 × 18 × 3 × 3 = 11,340 rows — no sampling needed)
- Assign `ticket_id` and `priority` to every row
- Save as `data/grid.csv` (the "skeleton" — no ticket text yet)

### Step 2 — Generate Ticket Text (`generate.py`)
- Read `data/grid.csv`
- Chunk rows into batches of 5
- For each batch, build the prompt and call the Gemini API
- Parse the returned JSON array and write `ticket_text` back to each row
- Save incrementally to `data/tickets_raw.csv` (so progress is not lost on failure)

### Step 3 — Validate & Clean (`validate.py`)
- Drop any rows where `ticket_text` is empty or malformed
- Assert no exact duplicate `ticket_text` values
- Check priority class distribution matches targets (warn if off by >5%)
- Save final output to `data/tickets.csv`

## Downstream Use

`tickets.csv` feeds into `scripts/train_model.py`. The model is trained to predict `priority` from `ticket_text`.

Fine-tuning target: `distilbert-base-uncased` via Hugging Face `transformers` + `Trainer` API, replacing the current TF-IDF + Logistic Regression baseline.

## Testing Strategy

Each pipeline step has a corresponding test file under `tests/`. Tests for a step **must pass before moving on to the next step**.

| Test File | What It Covers |
|---|---|
| `test_priority_rules.py` | Every cell of the sentiment × complexity → priority matrix; noise perturbation stays within ±1 level and hits ~7% of rows |
| `test_build_grid.py` | Grid has exactly 11,340 rows; all expected columns present; no nulls; all categorical values are within the defined taxonomy; `priority` column matches the deterministic rules (before noise) |
| `test_generate.py` | Prompt template renders correctly for sample metadata; JSON parsing handles well-formed arrays and gracefully rejects malformed output; batch size boundaries (partial final batch) |
| `test_validate.py` | Drops rows with empty/malformed `ticket_text`; detects and removes exact duplicates; priority distribution warnings fire when off by >5% |

Run tests with: `python -m pytest synthetic_data/tests/ -v`
